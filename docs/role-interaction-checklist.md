# OctoAcme Role Interaction Checklist

Use this checklist during project planning, kickoff, and sprint ceremonies to ensure clarity on roles, responsibilities, and communication patterns.

## Pre-Project Planning Checklist

### Define Core Team Roles
- [ ] **Project Manager assigned** — Confirm PM name, contact info, and communication preferences
- [ ] **Product Manager assigned** — Confirm PdM name and decision-making authority
- [ ] **Technical Lead identified** — Confirm architecture owner and design review process
- [ ] **Engineering Manager informed** — Notify EM of team assignments and capacity constraints
- [ ] **Delivery Lead / Scrum Master assigned** — Confirm ceremony facilitation and blocker escalation
- [ ] **UX/Design resources identified** — Confirm design involvement and research timeline
- [ ] **Support/Customer Liaison identified** — Confirm customer context provider and escalation path

### Clarify Cross-Role Dependencies
- [ ] **PM-PdM alignment** — Confirm weekly sync time and decision-making protocol
- [ ] **PM-Tech Lead alignment** — Confirm technical feasibility review cadence
- [ ] **PdM-Designer alignment** — Confirm design discovery and wireframe timeline
- [ ] **EM-PM capacity discussion** — Confirm team capacity, constraints, and buffer for unplanned work
- [ ] **Delivery Lead-Team preparation** — Confirm sprint length, ceremonies, and metrics tracking

### Customer and Stakeholder Communication
- [ ] **Support/CS Liaison briefing** — Brief on project scope, success metrics, and escalation paths
- [ ] **Stakeholder list curated** — Confirm decision-makers and update frequency
- [ ] **Communication cadence documented** — Weekly standup, monthly stakeholder updates, etc.

---

## Sprint Planning Checklist

### Confirm Role-Specific Activities
- [ ] **PdM prioritized backlog** — Backlog sorted by value, includes acceptance criteria
- [ ] **Tech Lead reviewed for feasibility** — Design review completed, risks flagged
- [ ] **Designer prepared specs** — Wireframes, acceptance criteria, and edge cases defined
- [ ] **Support/CS Liaison reviewed** — Customer impact and support scenarios considered
- [ ] **EM confirmed capacity** — Team capacity allocated, no overallocation
- [ ] **Delivery Lead prepared ceremonies** — Sprint goals defined, blockers pre-identified

### Team Readiness
- [ ] **Developers understand acceptance criteria** — Specs reviewed, design walkthrough scheduled
- [ ] **QA aligned on test scenarios** — Edge cases, smoke tests, and support scenarios identified
- [ ] **Technical Lead available for design discussions** — Architecture decisions documented

---

## Daily Standup Role Verification

### Confirm Role Participation
- [ ] **Developers report progress and blockers**
- [ ] **Tech Lead flags architectural or design risks**
- [ ] **Delivery Lead captures blockers and dependencies**
- [ ] **QA confirms readiness and test progress**
- [ ] **Support Liaison present for customer-impacting work**

### Escalation Path Clarity
- [ ] **Blockers escalated to PM if cross-team** — PM coordinates dependency resolution
- [ ] **Technical risks escalated to Tech Lead** — Architecture or design decisions needed
- [ ] **Support concerns escalated to Delivery Lead** — Customer impact flagged for priority discussion

---

## Sprint Review / Demo Checklist

### Role-Specific Participation
- [ ] **Product Manager or Customer present** — Validates acceptance criteria met
- [ ] **Stakeholders invited** — Key decision-makers attend
- [ ] **Support/CS Liaison present** — Provides feedback on usability and support readiness
- [ ] **UX/Designer present** — Validates design implementation quality
- [ ] **Team presents findings** — Demos, metrics, and learnings shared

### Acceptance Criteria Validation
- [ ] **Functional requirements met** — Feature works as specified
- [ ] **Design quality validated** — Matches design system and UX standards
- [ ] **Support/operations readiness** — Runbooks, debugging, customer docs prepared
- [ ] **Performance acceptable** — Load, latency, and resource metrics acceptable

---

## Sprint Retrospective Checklist

### Inclusive Participation
- [ ] **All team roles present** — PM, PdM, EM, Developers, Tech Lead, Delivery Lead, QA, Support
- [ ] **Psychological safety fostered** — Anonymous ideas encouraged if needed
- [ ] **Blameless culture emphasized** — Focus on process improvement, not blame

### Role-Specific Retrospective Questions
- [ ] **Tech Lead**: Were technical decisions made quickly? Was design review efficient?
- [ ] **EM**: Was team workload balanced? Did anyone feel overloaded or blocked?
- [ ] **Delivery Lead**: Were ceremonies effective? Were blockers addressed quickly?
- [ ] **Developers**: Were acceptance criteria clear? Did tools/infrastructure support us?
- [ ] **QA**: Were test scenarios well-defined? Did we find issues early?
- [ ] **Support/CS**: Would this feature be easy to support? Were customer needs addressed?
- [ ] **PdM**: Did we deliver the right thing? Did we learn from the market?
- [ ] **PM**: Were dependencies managed well? Was communication clear?

### Action Items Ownership
- [ ] **Clear owners assigned** — Each action item has a named owner and due date
- [ ] **Success criteria defined** — How will we know the action item is complete?
- [ ] **EM support identified** — Does EM need to support action (hiring, training, etc.)?

## Release Readiness Checklist

### Cross-Role Sign-Off
- [ ] **PM approval** — Schedule and go/no-go decision
- [ ] **Tech Lead sign-off** — Architecture sound, no known technical debt blockers
- [ ] **EM confirms team readiness** — Team availability, on-call support arranged
- [ ] **Delivery Lead confirms process** — Release runbook reviewed, rollback tested
- [ ] **QA confirms quality gates** — Security scan, smoke tests, regression tests passed
- [ ] **Support/CS Liaison confirms readiness** — Runbooks written, support team trained
- [ ] **PdM confirms success metrics** — Measurement tools in place, success criteria documented
- [ ] **Designer/UX confirms user experience** — Edge cases tested, accessibility verified

### Communication Readiness
- [ ] **Release notes drafted** — Feature summary, migration steps, known issues documented
- [ ] **Stakeholder notification planned** — Email, announcement, or briefing scheduled
- [ ] **Support team briefing completed** — FAQs, troubleshooting guide, escalation paths reviewed
- [ ] **Post-release metrics dashboard ready** — Error rates, performance, usage tracking enabled

---

## Post-Release / Incident Response Checklist

### Role-Specific Responsibilities
- [ ] **On-call Engineer responds** — Initial triage within SLA
- [ ] **Tech Lead engages** — Provides architectural context, mitigation options
- [ ] **PM coordinates escalation** — Notifies stakeholders, updates communication plan
- [ ] **Support Liaison alerts customers** — Provides status updates, collects impact info
- [ ] **EM ensures team support** — Team wellbeing, off-hours support coordination
- [ ] **Delivery Lead tracks incident** — Timeline, action items, blameless retrospective scheduled

### Post-Incident Review
- [ ] **Blameless retrospective scheduled** — Within 48 hours of resolution
- [ ] **Root cause analysis** — What happened, why, and what systems failed
- [ ] **Action items assigned** — Preventive measures, monitoring improvements, runbook updates
- [ ] **Learning captured** — Knowledge base updated for future incidents

---

## How to Use This Checklist

1. **Pre-project**: Run the pre-project planning checklist to ensure all roles are clear and aligned
2. **Ongoing**: Use relevant sections during each sprint cycle (planning, daily standups, review, retro)
3. **Release**: Use the release readiness checklist before deploying to production
4. **Incidents**: Use the incident response section for rapid coordination during outages
5. **Retrospectives**: Reference role-specific retro questions to ensure all perspectives are heard

---

## Notes

- Modify this checklist based on your team structure. Not all roles may be present on every project.
- Use this as a discussion guide during retrospectives to identify gaps in communication or process.
- Update this document when new roles are added or when you discover process improvements.
